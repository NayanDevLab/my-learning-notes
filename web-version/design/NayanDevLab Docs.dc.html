🔀Merge Sorted Array

Merge Sorted Array (LeetCode 88) asks you to merge two sorted integer arrays, nums1 and nums2, into one sorted array — in place, reusing the spare capacity already at the end of nums1. It is the classic warm-up for the two-pointer technique.

🤔Understanding the problem

The tricky part is not the sorting — both inputs arrive already sorted. It is doing the merge without a second buffer. A few details decide the whole approach:

  • nums1 has length m + n; its last n slots are zeros — scratch space, not real values.
  • The result must stay sorted, and we must not overwrite a value in nums1 before we have read it:
    • Writing front-to-back clobbers unread elements.
    • Writing back-to-front never does — that is the key insight.
  • Return nothing; the mutation happens inside nums1.
💡
Tip

When an array hands you trailing scratch space, read it as a hint to fill from the back. Spotting that spare capacity is half of solving the problem.

🎯
Key point

Hold one invariant: the write index k never moves past a value you haven't read yet. Keep that true and correctness follows for free.

✅The two-pointer approach

Walk both arrays from the end and fill nums1 from its last index backward. At each step, copy whichever tail value is larger:

  1. 1. Set i = m - 1, j = n - 1, and a write pointer k = m + n - 1.
  2. 2. Compare nums1[i] with nums2[j]; place the larger at nums1[k] and step that pointer back.
    • If nums1 runs out first, keep draining nums2.
  3. 3. Stop once nums2 is fully merged — any remaining nums1 values are already in place.
⚠️
Gotcha

Don't merge front-to-back. Writing from index 0 overwrites nums1 values you haven't compared yet — the classic wrong turn on this problem.

Because we only ever write to slots at or beyond k, and k always stays ahead of both read pointers, we never overwrite a value we still need.

Trace it on a tiny example — nums1 = [1,2,3,0,0,0], nums2 = [2,5,6] — to lock in the invariant:

Dry run walking i, j, k from the back 
idx │  i  j  k │ compare     │ write       │ nums1
────┼──────────┼─────────────┼─────────────┼──────────────────
  0 │  2  2  5 │ 3 < 6       │ nums1[5]=6  │ [1 2 3 0 0 6]
  1 │  2  1  4 │ 3 < 5       │ nums1[4]=5  │ [1 2 3 0 5 6]
  2 │  2  0  3 │ 3 > 2       │ nums1[3]=3  │ [1 2 3 3 5 6]
  3 │  1  0  2 │ 2 = 2       │ nums1[2]=2  │ [1 2 2 3 5 6]
  4 │  1 -1  1 │ j < 0: stop │ —           │ [1 2 2 3 5 6] ✓

⭐Complexity & common mistakes

Every element is visited once and no extra array is allocated — time O(m + n), space O(1). Most bugs come from merging the wrong direction:

⚡
Complexity

Both read pointers and the write pointer only ever move backward, so the loop runs at most m + n times — no nested scan hides anywhere. Time O(m + n), space O(1).

❌  Instead of this ✅  Do this
Allocate a third array, merge, then copy back Merge in place, writing from the back
Compare from the front (index 0) Compare from the end (m-1, n-1)
Loop while i >= 0 && j >= 0 Loop while j >= 0 — the rest is sorted
Treat trailing zeros as real data Treat slots m … m+n-1 as writable space
✅
Why it's optimal

You must touch every element at least once just to produce the merged output, so O(m + n) time is a hard floor — and reusing the trailing space keeps it at O(1) extra. This solution can't be beaten asymptotically.

Before marking this one as learned:

  • Handle the edge case where nums1 is empty (m = 0)

  • Explain out loud why back-to-front avoids a buffer

🧠Key takeaways

When an array has trailing space and you need an in-place merge, fill from the back. The same backward-write trick reappears in:

  • String problems that grow a buffer in place.
  • Any “merge two sorted X” variant under tight memory limits — see the Two Pointers note.
🌶️
Bonus

The same back-to-front fill generalizes: to merge k sorted arrays, keep the current tails in a max-heap and pop the largest each step — O((m + n) log k), still writing in place.
